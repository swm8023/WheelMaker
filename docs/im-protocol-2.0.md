# IM Protocol 2.0

Updated: 2026-04-07
Status: Draft (Protocol Freeze Candidate)

## 0. Purpose

IM Protocol 2.0 defines a chat-centric routing protocol between IM channels and one WheelMaker client runtime.

This protocol standardizes:

1. Chat identity (`routeKey`)
2. Route identity (`routeKey = imType + ":" + chatID`)
3. Session binding (`clientSessionId`)
4. Inbound/outbound event normalization
5. Chat binding persistence in IM domain

## 1. Hard Rules

1. One `Client` MUST own exactly one `IMRouter`.
2. `IMRouter` MUST be ACP-transparent; it cannot parse ACP business details.
3. One `clientSessionId` MAY bind to multiple active chats.
4. One active chat MUST bind to exactly one `clientSessionId` at any time.
5. IM protocol MUST NOT include `userId` as a required field.
6. IM persistence MUST only store IM-chat projection data; client-owned session lifecycle data is out of scope.
7. Routing identity MUST directly use `routeKey` (`imType:chatID`), with no extra route indirection.

## 2. Terms

- `clientSessionId`: persistent virtual session identity in client domain.
- `IMType`: IM adapter/channel type key (for example `feishu`, `console`).
- `chatID`: channel-native chat identifier generated by IM channel itself.
- `routeKey`: normalized endpoint id `<imType>:<chatID>`, used directly for routing.
- `IMActiveChat`: lightweight runtime+persisted projection of one chat endpoint.
- `IMRouter`: single ingress/egress entry for IM events of one client.

## 3. Event Model

### 3.1 Inbound (IM -> Client)

Inbound kinds:

1. `prompt`
2. `permission_reply`
3. `command`

Required fields:

- `kind`
- `imType`
- `chatID` or `routeKey`
- `receivedAt`

Router normalization behavior:

1. Resolve `routeKey` (build from `imType+chatID` when missing).
2. Resolve `clientSessionId` by `routeKey` binding.
3. If binding does not exist: create new `clientSessionId` and bind.
4. Dispatch normalized event to client callback with both `routeKey` and `clientSessionId`.

### 3.2 Outbound (Client -> IM)

Outbound kinds:

1. `message`
2. `acp_update`
3. `command_reply`

Required fields:

- `kind`
- `clientSessionId`

Optional fields:

- `targetRouteKey` (empty means broadcast)
- `text`
- `payload`
- `meta`

Router send behavior:

1. If `targetRouteKey` is set: targeted send to one chat channel.
2. If `targetRouteKey` is empty: broadcast to all online chats bound to `clientSessionId`.

## 4. Session Binding Semantics

### 4.1 Lazy Session Bootstrap

For first message from a new chat:

1. Router creates `clientSessionId` via callback.
2. Router persists binding `routeKey -> clientSessionId`.
3. Router forwards normalized inbound event to client.
4. Client may lazily `load/new` ACP session later.

### 4.2 `/new` Rebind Scope

`/new` triggered from one chat MUST only affect that chat binding:

1. Client creates a new `clientSessionId`.
2. Router rebinds that chat to the new session id.
3. Other chats remain bound to original session.

### 4.3 Source-aware Reply

Client SHOULD send reply with `targetRouteKey` equal to inbound source chat when reply is chat-specific.

Broadcast remains default when target is omitted.

## 5. ACP Transparency

`acp_update` payload is opaque in IM layer.

- Router does not inspect ACP update schema.
- IM channels decide rendering.

## 6. Persistence Model (IM Domain Only)

Storage scope: chat projection only.

Table: `im_route_bindings`

- `project_name` TEXT NOT NULL
- `route_key` TEXT NOT NULL
- `im_type` TEXT NOT NULL
- `chat_id` TEXT NOT NULL
- `client_session_id` TEXT NOT NULL
- `online` INTEGER NOT NULL
- `last_seen_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

Primary Key: `(project_name, route_key)`

Recommended indexes:

1. `(project_name, client_session_id)`
2. `(project_name, online)`

## 7. Write Policy

Hybrid write policy:

1. Critical sync write:
   - first bind of a new chat
   - explicit rebind (for example `/new` on current chat)
2. Debounced async write:
   - online/offline updates
   - heartbeat/last seen touch

## 8. Errors and Compatibility

1. Unknown `imType` channel during outbound send: return error to caller.
2. Missing binding on inbound: create binding instead of dropping message.
3. Empty outbound target set with no online chats: no-op, return nil.
4. Protocol is designed to coexist with legacy IM flow during migration.
